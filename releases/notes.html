RETRIEVAL METHODOLOGY

Tools

Tools in the MetricsGrimoire toolset have been used to retrieve the repositories with the source data for the analysis. In particular, the following tools have been used.

  • CVSAnalY

    This tool has been used to retrieve data about changes to the source code from the git repositories, and store it in a MySQL database.

  • Bicho

    This tool has been used to retrieve data from tickets in the the Launchpad repository, and store it in a MySQL database.

  • CVSAnalY

    This tool has been used to retrieve data from the OpenStack MailMan repositories, and store it in a MySQL database.

The resulting database (sanitized to remove email addresses) is available for download .

Extension

The retrieved data spans for all the history of OpenStack up to April 3rd 2013. The retrieval was performed on this latter date. However, the data shown corresponds only to the Grizzly release cycle, since Sept. 27th 2012.

FILTERING METHODOLOGY

Bots

Many commits in OpenStack are actually performed by bots, such as Gerrit, in charge of finishing code review processes. For the analysis of companies, we have ignored those commits.

In addition, data presented for the issue tracking system and mailing lists is based only on developers found in the versioning system. In other words, this analysis shows information about developers activity (and their affiliations) through the three data sources. As an example, a developer of a given company with no activity in the Git is only considered for the global numbers, but not for the specific ones of such company.

ANALYSIS METHODOLOGY

Techniques and details used in the analysis of data.

Affiliation

Affiliation of developers to companies and other institutions has been done based on various heuristics, some of the automatic, some other manual. The base heuristic has bee using the domain part of the email address to identify an institution, and affiliation of the corresponding developer to it. We have also used some information provided by the OpenStack Foundation, and a git-dm config file used by OpenStack developers to do a similar analysis. We have complemented and validated this information in many cases by manual search in the Internet. As a result of this process, we have classified all developers as affiliated with an organization (which we refer in this report as "company"), as non-affiliated , or as unknown (for which we couldn't decide to which company they were affiliated). We have also considered that developers may move from company to company, identifying for some of them the first or last date of work for their employers.

Non-affiliated

After doing some research, we have found some developers to be working on their own (be them volunteers or not). We consider them as non-affiliated. They have been found to have very little activity in total, compared to corporate activity.

Unknown affiliation

For some developers, we couldn't decide if they were affiliated to some company, or to which company in particular they were affiliated. We consider them as "with unknown affiliation".

Identification of unique developers

We have integrated, identifying duplicate identities of many (hopefully all relevant) developers.

VISUALIZATION

The visualization of the data has been produced using the vizGrimoire toolset. In particular the JavaScript producing the charts and tables is a part of the vizGrimoire.JS system. It visualized data present in JSON files produced by vizGrimoire.R from the MySQL database where all the base information is stored.

CHARTS AND TABLES

Summary charts

Charts that offer a summary, or an overview, of certain aspects of the project.

  • Commits per company (weekly)

    For each company, it shows the number of commits per month for authors affiliated to that company. Only the top ten companies, by total number of commits, are shown.

  • Commits per company (aggregated)

    For each company, it shows the number of commits for authors affiliated to that company. Only top ten companies, by total number of commits, are shown. Commits corresponding to authors for which we have not identified an affiliation are not considered.

  • Closed tickets per company (weekly)

    For each company, it shows the number of the evolution of closed tickets. Only top ten companies, by total number of closed issues, are shown. We count an issue as closed when the final stage is 'Fix Committed'.

  • Closed tickets per company (aggregated)

    For each company, it shows the number of closed issues, for authors affiliated to that company. Only top ten companies, by total number of closed issues, are shown. Issues corresponding to authors for which we have not identified an affiliation are not considered.

  • Sent messages per company (weekly)

    For each company, it shows the evolution of the number of messages sent by its developers.

  • Sent messages per company (aggregated)

    For each company, it shows the number of total messages sent for the specified period.

  • Top contributors tables

    Top authors, ordered by number of commits, messages sent and closed issues for the specified period.

  • Analysis of Companies

    List of top ten companies with information about commits, authors, touched files, added lines, removed lines and the addition of added and removed lines. This analysis provides information about core and non-core projects. Core projects are Nova, Swift, Glance, Keystone, Horizon, Quantum and Cinder.

Information for all companies

Charts offering an aggregated view of the data, organized by company. Three main pages are described for each of the data sources: versioning system, ticketing system and mailing lists.

  • Stacked chart of activity (evolution chart)

    Information about the evolution of the number of active companies in the give metric (commits, messages sent or closed bugs). We consider a company as active during a certain period when at least a developer affiliated to that company performed a commit, sent a message or closed an issue during that period.

  • Reviewed commits per company (aggregated) (bar chart)

    Aggregated (for the whole history of the project) number of reviewed commits per company. We assign a reviewed commit to a company when its author is a developer affiliated to that company. Only top ten companies are included (the remaining commits are not presented in this chart).

  • Activity per company (aggregated)

    Aggregated activity (for the whole history of the project) number of authors per company. We assign authors to a company when they are affiliated to that company. Only top ten companies are included (the remaining authors are not presented in this chart).

  • Activity per company (aggregated)

    Aggregated number of authors per company commiting changes, sending messages or closing issues. We assign authors to a company when they are affiliated to that company. Only top ten companies are included (the remaining authors are not presented in this chart).

Information per company

Charts and tables that offer several metrics for a given company.

Information is structured showing information about the evolution of a specific metric. The left side of the dashboard shows static or aggregated information about each of the three data sources. In the middle specific charts can be found about the evolution of specific metrics such as commits, authors, messages sent or closed issues. And finally, tables found at the right part show information about top contributors per company for each of the data sources.
